# Proposal Template

> This template defines the structure for `Proposal.md` generated by paperclaw-ideation-AI Phase 4.
> The ideation pipeline generates the proposal content; this file provides the skeleton.

---

## Header

```markdown
# Research Proposal: <Title>

> Generated by paperclaw-ideation-AI
> Date: <timestamp>
> Target Venue: <venue from domain.md>
```

---

## Section 1: Research Background

Why this problem matters. Real-world or scientific significance, broader context, application domains affected, and why now is the right time.

Ground every claim in literature. Target: 2-3 paragraphs.

---

## Section 2: Research Problem

Formal problem definition:
- Intuitive description
- Input space, output space, and objective
- Mathematical formalization (optimization, learning, inference)
- What constitutes success
- Key technical challenges

Target: 1-2 paragraphs of intuition + formal mathematical definition.

---

## Section 3: Related Work

Structured survey organized by method category. Use tables, not prose paragraphs.

### 3.1 [Category Name 1]
| Venue/Year | Paper | Technical Summary | Gap for Our Problem |
|-----------|-------|-------------------|---------------------|
| ... | ... | 1-2 sentences | What limitation remains |

### 3.2 [Category Name 2]
| Venue/Year | Paper | Technical Summary | Gap for Our Problem |
|-----------|-------|-------------------|---------------------|
| ... | ... | ... | ... |

(Repeat for 2-4 method categories, 3-8 papers per table.)

### Research Gap Summary

One paragraph synthesizing gaps from all tables into the specific niche this proposal fills.

---

## Section 4: Theoretical Analysis

### 4.1 Why This Theory Is Needed

What limitation this theory addresses, why theoretical foundation (not just empirical) is necessary, what would go wrong without it.

### 4.2 Mathematical Foundation

Complete notation, input/output spaces, objective function, constraints, assumptions, key definitions.

### 4.3 Main Theoretical Results

For EACH theorem/proposition:

**Theorem N: [Name]**

- **Assumptions:** Numbered A1, A2, ... with standard/novel classification.
- **Statement:** Formal mathematical statement, self-contained.
- **Proof Outline:** 3-5 step high-level strategy (always visible).
- **Detailed Proof:** Complete step-by-step proof. All intermediate steps justified.
  - In Proposal.md: inline, fully expanded.
  - In HTML: inside collapsible `<details>` block, default collapsed.
- **Lean 4 Verification:**
  1. Status: FULL PASS / PARTIAL PASS / FAIL
  2. Complete Lean 4 source code (full `.lean` file, not summary)
  3. Verification log (`lake build` output)
  4. Sorry items (if any, with justification)
  5. Gap analysis (if PARTIAL/FAIL)
- **Implications:** What this result means for the method; connection to gaps from Section 3.

Target: 1-3 theorems with COMPLETE proofs and COMPLETE Lean 4 code.

### 4.4 Verification Summary

| Theorem | Formalizable? | Lean 4 Status | Sorry Count | Key Insight |
|---------|--------------|---------------|-------------|-------------|
| Theorem 1: [Name] | Yes/Partially | PASS/PARTIAL/FAIL | N | [what verification revealed] |

If claims were not formalized, explain WHY for each one.

---

## Section 5: Proposed Method

### 5.1 Overview

High-level overview: architecture diagram (Mermaid), algorithm flow, pseudocode.

### 5.2 Component Details

For EACH key component:
1. **What**: Technical description
2. **Why this design**: What limitation motivated this choice
3. **Why not alternatives**: Simpler approaches considered and why insufficient
4. **Advantage**: Concrete benefit (theoretical guarantee, efficiency, robustness)

### 5.3 Training / Inference Procedure

Step-by-step procedure with implementation detail. Key hyperparameters and roles.

### 5.4 Design Rationale Summary

| Component | Addresses Limitation Of | Advantage Over Alternative |
|-----------|------------------------|---------------------------|
| [component 1] | [method X] lacks... | [concrete advantage] |

Target: 4-7 paragraphs + pseudocode + rationale table.
